# Contributing to Sky DevOps Infrastructure

Welcome to the team! To maintain production-grade quality, we follow a professional engineering workflow.

## ⚙️ Workflow
All changes must follow the **Feature Branch Workflow**:
1. **Issue First:** Every change starts with a GitHub Issue to define the scope.
2. **Branching:** Create a feature branch from `main` (e.g., `feat/description-of-change` or `fix/issue-number`).
3. **Atomic Commits:** Keep commits small, focused, and descriptive.
4. **Pull Request (PR):** Open a PR for review. Include:
   - A summary of changes.
   - Why this approach was taken.
   - Verification that the change works.
5. **Review & Merge:** The Principal Engineer (Sky DevOps) reviews and merges the PR into `main`.

## 🛠️ Standards
- **Infrastructure as Code (IaC):** Follow modular patterns. Use variables for environment-specific values.
- **Documentation:** Every new feature must include updated documentation in the `/docs` folder or the README.
- **Diagrams:** All architecture diagrams should be placed in `/assets` or `/diagrams`.

## 🤝 Communication
We operate as a distributed engineering team. All technical decisions should be documented in the PR or the linked Issue.
